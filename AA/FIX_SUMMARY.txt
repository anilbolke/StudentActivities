                    DATABASE CONNECTION LIFECYCLE FIX
═══════════════════════════════════════════════════════════════════════════════

ERROR: java.sql.SQLNonTransientConnectionException: 
       No operations allowed after connection closed

LOCATION: QuestionUploadService.java:108 (conn.commit())

ROOT CAUSE: 
  DAOs were closing the database connection after each operation using
  try-with-resources. This broke transaction management because the
  QuestionUploadService needed the same connection to stay open across
  multiple DAO calls.

SOLUTION:
  Modified 3 Upload DAOs to NOT close connections in try-with-resources.
  Only PreparedStatements are closed now. Connection lifecycle is managed
  by QuestionUploadService (the transaction owner).

CHANGE PATTERN:

BEFORE (BROKEN):
  try (Connection conn = DatabaseConnection.getInstance().getConnection();
       PreparedStatement pstmt = conn.prepareStatement(sql)) {
      // execute query
  }  // <- Closes both connection AND statement

AFTER (FIXED):
  Connection conn = DatabaseConnection.getInstance().getConnection();
  try (PreparedStatement pstmt = conn.prepareStatement(sql)) {
      // execute query
  }  // <- Closes only statement, connection stays open

KEY INSIGHT:

When using a singleton database connection:
  ✅ DO close resources created by the DAO (statements, result sets)
  ❌ DON'T close the connection itself
  
The code that OPENS the connection should CLOSE it. Doing it the other
way breaks transaction management.
